Foundation

기초 다지기: 데이터베이스와 트랜잭션

이후 모든 내용의 토대가 되는 개념입니다. "데이터베이스가 뭔지는 대충 아는데 트랜잭션은 잘 모르겠다"면 이 섹션을 먼저 읽으세요. 특히 트랜잭션 개념은 뒤에 나오는 CDC, 웨어하우스의 ACID 지원, Lakehouse를 이해하는 데 꼭 필요합니다.

기초 개념

데이터베이스: 데이터를 구조적으로 보관하고 꺼내 쓰는 시스템

데이터베이스(DB)는 단순한 "데이터 저장 폴더"가 아닙니다. 여러 사람이 동시에 읽고 써도 데이터가 깨지지 않게 하고, 원하는 데이터를 빠르게 찾아주고, 장애가 나도 데이터를 잃지 않게 보장하는 소프트웨어 시스템입니다.

비유: 엑셀과 무엇이 다른가?

엑셀 파일을 100명이 동시에 열어서 수정한다고 상상해보세요. 누가 먼저 저장했는지에 따라 다른 사람의 수정이 사라집니다. 데이터베이스는 이 "동시 접근" 문제를 해결하도록 설계된 시스템입니다. 또한 엑셀은 100만 행이 넘으면 버티지 못하지만, DB는 수억 행에서도 원하는 행을 밀리초 안에 찾아냅니다.

  • 테이블(Table): 데이터를 행(row)과 열(column)로 정리한 표입니다. 예: orders 테이블에는 주문 하나가 한 행, 주문번호·금액·날짜가 각각 열입니다.
  • 스키마(Schema): 테이블의 구조 정의입니다. "주문 금액은 숫자여야 하고, 주문번호는 비어 있으면 안 된다" 같은 규칙을 강제합니다. 엑셀에는 없는 안전장치입니다.
  • SQL: DB에게 "무엇을 달라"고 말하는 표준 언어입니다. SELECT 금액 FROM orders WHERE 날짜 = 오늘처럼 선언적으로 질문합니다.
  • 인덱스(Index): 책 뒤의 찾아보기처럼, 특정 컬럼으로 빠르게 검색할 수 있게 만든 자료구조입니다. 인덱스가 없으면 DB는 전체 행을 처음부터 끝까지 훑어야 합니다.
핵심 개념

트랜잭션: "전부 성공하거나, 전부 없던 일이 되거나"

트랜잭션(Transaction)은 여러 개의 작업을 하나의 묶음으로 처리하는 단위입니다. 묶음 안의 작업이 모두 성공해야 결과가 저장되고(커밋), 하나라도 실패하면 전체가 취소됩니다(롤백).

비유: 계좌 이체

A가 B에게 10만 원을 보내는 작업은 사실 두 단계입니다. ① A 계좌에서 10만 원 차감, ② B 계좌에 10만 원 추가. 만약 ①만 실행되고 ②가 실패하면 10만 원이 증발합니다. 트랜잭션은 이 두 단계를 하나로 묶어 "둘 다 성공하거나 둘 다 취소"를 보장합니다. 이것이 은행 시스템이 DB 위에서 안전하게 돌아가는 이유입니다.

예시: 계좌 이체 트랜잭션 
BEGIN TRANSACTION;  -- 트랜잭션 시작: 여기서부터 하나의 묶음

UPDATE accounts SET balance = balance - 100000
WHERE account_id = 'A';   -- ① A 계좌에서 차감

UPDATE accounts SET balance = balance + 100000
WHERE account_id = 'B';   -- ② B 계좌에 추가

COMMIT;  -- 둘 다 성공했으니 확정. 실패했다면 ROLLBACK으로 전체 취소.
핵심 개념

ACID: 트랜잭션이 지키는 4가지 약속

트랜잭션을 지원하는 DB는 ACID라는 4가지 성질을 보장합니다. 뒤에서 "Lakehouse는 ACID 트랜잭션을 지원한다"는 문장이 나오는데, 그 의미가 바로 이것입니다.

성질뜻쉬운 설명
Atomicity
원자성		 전부 성공 또는 전부 취소 계좌 이체에서 "차감만 되고 입금은 안 됨"이 절대 일어나지 않습니다.
Consistency
일관성		 규칙을 어기는 상태가 저장되지 않음 "잔액은 음수가 될 수 없다"는 규칙이 있다면, 그 규칙을 깨는 트랜잭션은 거부됩니다.
Isolation
격리성		 동시에 실행돼도 서로 간섭하지 않음 두 사람이 마지막 남은 좌석 1개를 동시에 예약해도, 한 명만 성공합니다.
Durability
지속성		 커밋된 데이터는 장애가 나도 살아남음 결제 완료 직후 서버가 꺼져도, 재시작하면 결제 기록이 남아 있습니다.
연결 개념

트랜잭션 로그: DB의 일기장 (CDC의 출발점)

DB는 Durability(지속성)를 지키기 위해, 데이터를 실제로 바꾸기 전에 "무엇을 바꿀 것인지"를 먼저 로그 파일에 기록합니다. MySQL에서는 binlog, PostgreSQL에서는 WAL(Write-Ahead Log)이라 부릅니다.

비유: 가게 장부

가게 주인이 진열대 상품을 바꾸기 전에 장부에 "사과 3개 입고, 배 2개 판매"라고 먼저 적어두는 것과 같습니다. 진열대(실제 데이터)가 망가져도 장부(로그)만 있으면 처음부터 다시 재현할 수 있습니다.

왜 데이터 엔지니어링에서 중요한가

뒤에 나오는 CDC(변경 데이터 캡처)는 바로 이 트랜잭션 로그를 읽는 기술입니다. 운영 DB에 "최근 바뀐 거 뭐야?"라고 쿼리를 반복해서 묻는 대신, DB가 이미 쓰고 있는 일기장을 옆에서 같이 읽는 것입니다. 그래서 운영 DB에 부담을 거의 주지 않고 모든 변경(insert, update, delete)을 빠짐없이 가져올 수 있습니다.

핵심 개념

OLTP vs OLAP: 같은 DB라도 목적이 다르다

데이터베이스는 용도에 따라 크게 두 종류로 나뉩니다. 이 구분을 알면 "왜 분석할 때 운영 DB를 직접 쓰면 안 되는지", "왜 웨어하우스라는 별도 저장소가 필요한지"가 명확해집니다.

비교 항목OLTP (운영 처리)OLAP (분석 처리)
목적서비스 운영 (주문 생성, 로그인, 결제)분석과 의사결정 (월 매출 집계, 추세 분석)
쿼리 형태"주문번호 1234 조회" — 소수 행을 빠르게"최근 1년 월별 매출" — 수백만 행을 집계
처리 단위초당 수천 건의 작은 트랜잭션가끔 실행되는 무거운 분석 쿼리
저장 방식행 기반 (Row Store)열 기반 (Column Store)
대표 제품MySQL, PostgreSQL, OracleSnowflake, BigQuery, Redshift
⚠️ 이것이 데이터 엔지니어링이 존재하는 이유 중 하나

분석가가 OLTP 운영 DB에 무거운 집계 쿼리를 직접 날리면, 그 쿼리가 도는 동안 실제 고객의 주문 처리가 느려질 수 있습니다. 그래서 운영 DB의 데이터를 분석 전용 저장소(OLAP)로 복제해 옮기는 파이프라인이 필요하고, 그것이 다음 섹션부터 설명하는 라이프사이클의 출발점입니다.

Core Framework

데이터 엔지니어링 라이프사이클

책 전체를 관통하는 핵심 프레임워크입니다. 데이터는 "생성 → 저장 → 수집 → 변환 → 서빙"의 단계를 거칩니다. 각 단계는 독립적이지 않고 서로 의존하며, 어느 한 곳이 약하면 다음 단계가 무너집니다.

Generation
데이터 생성

데이터는 업무가 일어날 때 자연스럽게 생깁니다

고객이 앱에서 버튼을 클릭하거나, 결제를 완료하거나, 영업 담당자가 CRM을 업데이트하면 데이터가 기록됩니다. 이 단계의 데이터는 원본 그대로여서 불완전하거나 중복될 수 있습니다. 데이터 엔지니어는 이 원본을 직접 만들지 않고, 잘 받아올 수 있는 구조를 설계합니다.

Storage
저장

어디에, 어떤 형태로 두느냐가 이후 모든 것을 결정합니다

파일 그대로 두는 것(Data Lake), SQL로 바로 조회 가능하게 구조화하는 것(Data Warehouse), 두 방식을 결합한 것(Lakehouse)이 있습니다. 저장 방식은 비용, 접근 속도, 쿼리 편의성에 직접 영향을 줍니다.

Ingestion
수집

여러 소스에서 데이터를 안정적으로 옮겨오는 일

한 번에 대량으로 가져오거나(Batch), 실시간으로 흘려보내거나(Streaming), DB 변경분만 따라가는(CDC) 방식이 있습니다. 어느 방식이든 누락, 중복, 순서 뒤바뀜을 다뤄야 합니다.

Transformation
변환

원본 데이터를 "회사가 쓸 수 있는 언어"로 바꾸는 과정

원본은 컬럼 이름이 `usr_id_v2`처럼 불명확하고, NULL이 섞여 있고, 여러 테이블에 흩어져 있습니다. 변환은 이것을 "지난달 활성 사용자 수" 같은 비즈니스 질문에 바로 답할 수 있는 형태로 만드는 작업입니다.

Serving
서빙

사람과 시스템이 소비할 수 있는 형태로 내보내기

대시보드(BI), 머신러닝 모델 입력값(Feature table), 업무 도구로의 동기화(Reverse ETL) 등 다양한 형태로 최종 소비자에게 전달됩니다. 저장에서 끝나지 않고 실제로 사용될 때 데이터의 가치가 실현됩니다.

⚠️ 흔한 오해

데이터 엔지니어링은 "데이터 분석"이 아닙니다. 데이터 분석가가 질문을 던지고 답을 찾는다면, 데이터 엔지니어는 그 질문에 답할 수 있도록 데이터가 안정적으로 흐르는 파이프라인과 인프라를 만드는 사람입니다.

Step 01 · Generation

데이터 생성: 데이터는 어디서 처음 만들어지나

데이터 엔지니어링 라이프사이클은 데이터가 "생성되는 순간"에서 시작합니다. 이 단계를 이해하면 왜 데이터가 지저분하고 불완전한지, 왜 수집이 복잡한지를 납득할 수 있습니다.

OLTP DB

운영 데이터베이스

서비스가 실시간으로 사용하는 DB입니다. 주문, 결제, 계정, 재고 등이 여기에 기록됩니다. 빠른 읽기/쓰기에 최적화되어 있어 분석용 복잡 쿼리를 그대로 날리면 서비스 성능이 떨어집니다.

Event Logs

이벤트 로그

사용자가 앱에서 행동할 때 생기는 기록입니다. 버튼 클릭, 페이지 조회, 기능 사용이 모두 이벤트입니다. 양이 방대하고 중복·누락이 흔합니다. 이벤트 이름과 속성 정의를 잘 해야 분석 가치가 생깁니다.

External APIs

외부 SaaS API

Salesforce, Stripe, Zendesk처럼 외부 도구가 보관하는 데이터를 API로 가져옵니다. 호출 횟수 제한(Rate Limit), 페이지네이션, 인증 토큰 갱신이 수집의 복잡성을 높입니다.

중요 개념

소스 시스템의 특성을 이해해야 하는 이유

소스 시스템은 데이터 엔지니어가 통제할 수 없습니다. 운영팀이 DB 스키마를 바꾸면 파이프라인이 깨지고, 이벤트 이름이 변경되면 분석 지표가 망가집니다.

비유

소스 시스템은 원자재 공급처입니다. 제철소가 광산의 철광석 품질을 통제할 수 없듯이, 데이터 엔지니어는 소스가 뱉어내는 데이터의 구조나 타이밍을 마음대로 바꿀 수 없습니다. 그래서 "변화에 강한 파이프라인"을 설계하는 게 중요합니다.

  • 스키마 드리프트: 소스 DB의 컬럼이 추가·삭제·이름 변경되면 파이프라인이 깨집니다. 이를 감지하고 알림을 보내는 장치가 필요합니다.
  • 볼륨 급증: 이벤트 기반 마케팅 캠페인을 열면 로그 양이 수십 배 늘어날 수 있습니다. 파이프라인이 이 급증을 버텨야 합니다.
  • 소스 가용성: 외부 API가 다운되거나 DB에 락이 걸리면 수집이 실패합니다. 재시도와 실패 격리 설계가 필요합니다.
실제 예시: 전자상거래의 주문 데이터

사용자가 결제 버튼을 누르면 운영 DB에 주문 레코드가 생깁니다. 동시에 결제 게이트웨이 웹훅이 도착하고, 재고 시스템이 수량을 차감하고, 물류 시스템이 배송 요청을 만듭니다. 이 세 시스템의 데이터를 합쳐야 "실제 주문 현황"이 보입니다. 데이터 엔지니어의 일은 세 소스를 안정적으로 가져와 연결하는 파이프라인을 만드는 것입니다.

Step 02 · Storage

저장소: 데이터를 어디에 어떻게 두느냐

저장소 선택은 "무조건 더 좋은 것"이 없습니다. 어떤 분석을 얼마나 빠르게, 얼마의 비용으로 하느냐에 따라 적합한 저장 방식이 달라집니다.

Data Lake

데이터 레이크

원본 파일을 있는 그대로 큰 저장 공간에 모아두는 방식입니다. S3, GCS, ADLS 같은 오브젝트 스토리지가 대표적입니다. 저렴하고 유연하지만 관리가 없으면 "데이터 늪(Data Swamp)"이 됩니다.

Data Warehouse

데이터 웨어하우스

분석에 최적화된 구조화 저장소입니다. SQL로 쉽게 조회하고, 권한을 관리하고, 성능을 튜닝할 수 있습니다. Snowflake, BigQuery, Redshift가 대표적입니다. 비용이 레이크보다 높습니다.

Lakehouse

레이크하우스

레이크의 저렴함과 유연성 + 웨어하우스의 테이블 관리와 ACID 트랜잭션을 결합한 접근입니다. Delta Lake, Apache Iceberg, Apache Hudi가 이 범주에 속합니다. 운영 복잡도는 높습니다.

핵심 개념

Row Store vs Column Store

저장소가 데이터를 행(Row) 단위로 저장하느냐, 열(Column) 단위로 저장하느냐는 성능에 크게 영향을 줍니다.

행 기반 저장 (Row Store)

한 사람의 모든 정보(이름, 나이, 이메일, 주소)를 묶어 저장합니다. 특정 사람 한 명의 정보를 빠르게 읽거나 쓰는 데 좋습니다. MySQL, PostgreSQL 같은 운영 DB가 이 방식입니다.

열 기반 저장 (Column Store)

"이름" 컬럼 전체, "나이" 컬럼 전체처럼 컬럼 단위로 묶어 저장합니다. "전체 사용자의 평균 나이는?"처럼 특정 컬럼만 읽는 분석 쿼리가 훨씬 빠릅니다. Snowflake, BigQuery가 이 방식입니다.

중요 개념

파일 포맷: CSV vs Parquet vs JSON

데이터를 저장하는 파일 형식도 중요합니다. 분석 파이프라인에서는 CSV가 아니라 Parquet을 쓰는 이유가 있습니다.

  • CSV: 엑셀로 열 수 있고 사람이 읽기 좋습니다. 하지만 타입 정보가 없고, 큰 파일을 읽을 때 전체를 스캔해야 합니다. 데이터 교환용으로 적합하지만 분석 저장에는 비효율적입니다.
  • JSON: 중첩 구조(배열, 객체)를 표현하기 좋습니다. API 응답, 이벤트 로그에 많이 쓰입니다. 그러나 크기가 크고 분석 쿼리가 느립니다.
  • Parquet: 열 기반 저장을 파일 수준에서 구현합니다. 압축률이 높고 필요한 컬럼만 읽어 쿼리가 빠릅니다. 데이터 레이크와 웨어하우스 파이프라인의 표준 형식입니다.
예시: S3에 저장된 Parquet 파일을 Snowflake에서 외부 테이블로 조회 
-- S3의 원본 Parquet 파일을 Snowflake에서 직접 조회하는 외부 테이블 정의
CREATE EXTERNAL TABLE raw.product_events (
  event_id    VARCHAR,
  event_name  VARCHAR,
  user_id     VARCHAR,
  occurred_at TIMESTAMP_NTZ
)
WITH LOCATION = @raw_events_stage/'2024/'
FILE_FORMAT = (TYPE = 'PARQUET')
AUTO_REFRESH = TRUE;

-- 외부 테이블은 파일을 복사하지 않고 S3를 직접 참조합니다.
-- 비용은 저렴하지만 쿼리 성능은 웨어하우스 내 테이블보다 느립니다.

Step 03 · Ingestion

수집: 데이터를 소스에서 플랫폼으로 옮기기

수집은 단순한 복사가 아닙니다. 중복, 누락, 순서 뒤바뀜, API 제한, 네트워크 장애를 모두 고려해야 하며, "얼마나 빠르게, 얼마나 정확하게, 얼마의 비용으로"를 균형 맞춰야 합니다.

Batch

배치 수집

정해진 주기(매일 새벽 2시, 매시 정각 등)에 데이터를 묶어서 가져옵니다. 구현이 단순하고 비용을 예측하기 쉽습니다. 재무 리포트, 일별 KPI처럼 실시간성이 덜 중요한 워크로드에 적합합니다.

Streaming

스트리밍 수집

이벤트가 생기는 즉시 처리합니다. Kafka, Kinesis 같은 메시지 큐를 통해 데이터가 끊임없이 흐릅니다. 실시간 사기 탐지, 개인화 추천, 운영 모니터링에 필요하지만 운영 복잡도와 비용이 높습니다.

CDC

변경 데이터 캡처

DB의 트랜잭션 로그를 읽어 insert, update, delete 변경분만 스트리밍합니다. 운영 DB를 직접 복제하지 않고 변경 이력만 따라가 부하가 적습니다. Debezium이 대표적인 도구입니다.

깊이 이해하기

배치 vs 스트리밍: 무엇이 더 좋은가?

스트리밍이 항상 더 좋지 않습니다. 업무 요구사항에 맞는 방식을 선택하는 것이 중요합니다.

"우리도 실시간으로 해야 하지 않나요?"라는 질문에 무조건 스트리밍으로 답하는 것은 위험합니다. 스트리밍은 비용과 운영 부담이 크게 늘어납니다. 먼저 "이 데이터가 몇 분 늦으면 실제로 어떤 비즈니스 문제가 생기는가?"를 물어야 합니다.

핵심 개념

CDC (Change Data Capture) 상세

CDC는 운영 DB의 트랜잭션 로그(binlog/WAL — 기초 섹션에서 설명한 "DB의 일기장")를 읽어 변경 사항을 추출합니다. 운영 DB에 직접 SELECT 쿼리를 반복하는 것보다 훨씬 효율적입니다.

운영 DBMySQL, Postgres 트랜잭션 발생
트랜잭션 로그binlog / WAL 기록
CDC 커넥터Debezium이 로그 읽기
메시지 큐Kafka 토픽으로 발행
분석 저장소웨어하우스에 적재
  • Append-only 원칙: 원본 변경 이벤트는 삭제하거나 덮어쓰지 않고 그대로 쌓아둡니다. 나중에 특정 시점의 상태를 재현할 수 있습니다.
  • DELETE 처리: 운영 DB에서 행이 삭제되면 CDC는 "이 행이 삭제됐다"는 이벤트를 기록합니다. 분석 저장소에서는 deleted_at 타임스탬프를 찍는 방식으로 처리합니다.
  • 스키마 변경: 소스 테이블에 컬럼이 추가되면 CDC가 이를 감지해 다운스트림 테이블 스키마도 업데이트해야 합니다. 이 부분이 자주 파이프라인을 깨뜨립니다.
실제 예시: 전자상거래 주문 상태 추적

주문이 "결제 대기 → 결제 완료 → 배송 준비 → 배송 중 → 배송 완료"로 상태가 바뀔 때, 매번 전체 주문 테이블을 복사하는 것은 비효율적입니다. CDC는 상태가 바뀔 때마다 그 변경 이벤트만 전달해 분석 저장소에 최신 상태와 이력을 모두 보존합니다.

예시: Kafka 스트림에서 이벤트를 읽어 중복 제거 후 적재하는 패턴 
-- 스트리밍 수집 후 웨어하우스에 중복 제거하여 upsert하는 패턴
MERGE INTO clean.orders AS target
USING (
  SELECT
    order_id,
    status,
    updated_at,
    ROW_NUMBER() OVER (
      PARTITION BY order_id ORDER BY updated_at DESC
    ) AS rn
  FROM raw.orders_cdc_stream
) AS source
ON target.order_id = source.order_id AND source.rn = 1
WHEN MATCHED THEN UPDATE SET
  target.status = source.status,
  target.updated_at = source.updated_at
WHEN NOT MATCHED THEN INSERT
  (order_id, status, updated_at)
  VALUES (source.order_id, source.status, source.updated_at);

-- ROW_NUMBER()로 같은 order_id의 가장 최신 이벤트만 선택합니다.
-- MERGE로 이미 있으면 업데이트, 없으면 삽입합니다.

Step 04 · Transformation

변환과 모델링: 원본을 믿을 수 있는 데이터 제품으로

원본 데이터는 분석가가 바로 쓸 수 없습니다. 컬럼명은 `usr_id_v2`, 날짜 형식은 `20240101`, 상태값은 `1`처럼 불명확합니다. 변환은 이것을 "누구나 이해하고 믿는" 형태로 바꾸는 작업입니다.

핵심 개념

ETL vs ELT: 순서의 차이

데이터를 어디에서 변환하느냐의 차이입니다. 클라우드 웨어하우스 시대에는 ELT가 주류가 되었습니다.

ETL (Extract → Transform → Load)

소스에서 꺼내서(E), 파이프라인 서버에서 변환하고(T), 정제된 데이터만 저장소에 넣습니다(L). 웨어하우스가 비싸고 컴퓨팅이 제한적이던 시절의 방식입니다. 원본이 저장소에 남지 않습니다.

ELT (Extract → Load → Transform)

소스에서 꺼내서(E), 원본 그대로 저장소에 넣고(L), 저장소 안에서 SQL로 변환합니다(T). Snowflake, BigQuery처럼 강력한 컴퓨팅을 가진 현대 웨어하우스에서 효율적입니다. 원본이 보존됩니다.

핵심 개념

데이터 레이어: Raw → Staging → Mart

변환은 한 번에 완성되지 않습니다. 단계별 레이어를 통해 점진적으로 정제합니다.

Raw

원본 그대로 (절대 수정하지 않음)

소스에서 받은 데이터를 그대로 저장합니다. 이 레이어는 어떤 처리도 하지 않습니다. 나중에 변환 로직이 잘못됐을 때 원본으로 돌아가 재처리할 수 있는 안전망입니다. 컬럼명이 `usr_id_v2`여도, 날짜가 문자열이어도 그대로 보존합니다.

Staging

정제와 표준화 (1:1 변환)

각 소스 테이블에 1:1 대응하는 정제 테이블을 만듭니다. 컬럼명을 표준화하고(`usr_id_v2` → `user_id`), 날짜 타입을 변환하고, NULL을 정의에 맞게 처리합니다. 여러 소스를 합치거나 비즈니스 로직을 적용하지 않습니다.

Intermediate

조인과 집계 (비즈니스 로직 적용)

여러 스테이징 테이블을 조인하고, 비즈니스 규칙을 적용합니다. "유효한 구독을 가진 계정만", "환불을 제외한 매출만"처럼 조직의 정의를 반영합니다. 최종 마트에 쓰이는 재사용 가능한 빌딩 블록입니다.

Mart

최종 소비용 (팀이 공유하는 신뢰 가능한 테이블)

BI 도구, 분석가, 데이터 과학자가 실제로 쓰는 테이블입니다. `mart_active_users`, `mart_revenue`, `mart_churn_risk` 처럼 특정 비즈니스 질문에 최적화되어 있습니다. 한 번 정의된 마트는 모든 팀이 동일한 정의로 사용합니다.

핵심 개념

Grain: 테이블의 한 행이 무엇을 의미하는가

Grain(그레인)은 테이블에서 한 행(row)이 무엇을 나타내는지를 정의합니다. "사용자당 1행", "주문당 1행", "사용자-날짜 조합당 1행"처럼 명확해야 합니다.

⚠️ Grain이 섞이면 이중 집계가 발생합니다

주문 테이블에 "주문당 1행"이어야 하는데 주문의 각 아이템마다 1행이 섞이면, SUM(amount)를 하면 금액이 아이템 수만큼 부풀어 오릅니다. 테이블을 설계할 때 Grain을 먼저 명확히 정의하는 것이 중요합니다.

예시: dbt를 사용한 활성 사용자 마트 정의 
-- models/marts/mart_active_users.sql
-- Grain: user_id당 1행. 최근 30일 이내 로그인한 유료 사용자만 포함.

WITH paid_users AS (
  SELECT user_id, plan_type, subscription_start_date
  FROM {{ ref('stg_subscriptions') }}
  WHERE status = 'active'
    AND plan_type != 'free'
),

recent_logins AS (
  SELECT
    user_id,
    MAX(occurred_at) AS last_login_at,
    COUNT(*) AS login_count_30d
  FROM {{ ref('stg_login_events') }}
  WHERE occurred_at >= CURRENT_DATE - INTERVAL '30 days'
  GROUP BY user_id
)

SELECT
  p.user_id,
  p.plan_type,
  p.subscription_start_date,
  r.last_login_at,
  r.login_count_30d,
  -- 마지막 로그인이 30일 이내면 활성 사용자로 분류
  r.last_login_at IS NOT NULL AS is_active
FROM paid_users p
LEFT JOIN recent_logins r USING (user_id);

데이터 품질 테스트

변환 결과가 올바른지 자동으로 검사합니다. "user_id가 NULL이면 안 된다", "주문 금액이 음수면 안 된다", "상태값이 정해진 목록 안에 있어야 한다"처럼 규칙을 코드로 정의합니다. dbt가 이 테스트를 내장 지원합니다.

데이터 리니지

Step 05 · Serving

서빙: 데이터를 소비자에게 맞는 형태로 전달하기

변환까지 완성된 데이터를 어떻게 누구에게 전달하느냐가 서빙입니다. 같은 데이터를 CEO는 대시보드로, 분석가는 SQL로, ML 팀은 피처 테이블로, 영업팀은 CRM 안에서 봅니다.

Analytics / BI

분석과 대시보드

Tableau, Looker, Metabase 같은 BI 도구에 연결해 현업이 대시보드와 리포트를 봅니다. Semantic layer를 통해 "매출"의 정의가 모든 대시보드에서 동일하게 적용됩니다.

Machine Learning

ML 피처 서빙

모델 학습과 추론에 쓰이는 입력값(Feature)을 안정적으로 제공합니다. Feature Store는 이 피처를 한 곳에 관리해 여러 모델이 재사용하게 합니다. 학습 시점과 추론 시점 데이터의 불일치(Training-Serving Skew)를 방지하는 것이 핵심입니다.

Reverse ETL

운영 시스템으로 역전송

웨어하우스에서 계산한 "고객 위험 점수", "업셀 후보", "이탈 예측"을 Salesforce, HubSpot, Zendesk 같은 업무 도구로 다시 보냅니다. 데이터를 보는 것을 넘어 자동 행동으로 연결합니다.

심화 개념

Semantic Layer: 숫자의 정의를 한 곳에서 관리하기

같은 "월 매출"이지만 재무팀은 결제 완료 기준, 마케팅팀은 주문 생성 기준으로 계산하면 숫자가 달라집니다. Semantic Layer는 이 정의를 한 곳에 작성해 모든 BI 도구와 쿼리가 동일한 공식을 쓰도록 강제합니다.

비유: 회사 공용 용어 사전

"활성 사용자"가 무엇인지 팀마다 다르게 정의한다면, 회의마다 숫자 검증에 시간을 씁니다. Semantic Layer는 이 정의를 코드로 고정해 "정의가 맞는지" 논쟁 자체를 없앱니다.

  • LookML (Looker): BI 도구 레벨에서 지표와 차원을 정의합니다. "revenue는 항상 환불을 제외한다"를 한 번 정의하면 모든 리포트에 적용됩니다.
  • dbt Metrics: 변환 레이어에 지표 정의를 포함시킵니다. SQL, BI, API 모두 같은 정의를 참조합니다.
  • Cube.dev: 독립적인 Semantic Layer 서버로 여러 BI 도구에 단일 지표 정의를 제공합니다.
실제 예시: Reverse ETL로 CS팀에 고객 건강 점수 전달

데이터 팀이 웨어하우스에서 계산한 "고객 건강 점수(0–100)"를 Salesforce의 각 계정 필드에 자동으로 동기화합니다. CS 담당자는 Salesforce에서 자신이 담당하는 계정의 위험도를 실시간으로 보고, 낮은 점수의 계정을 우선 연락합니다. 데이터 팀이 직접 리포트를 뽑아주지 않아도 됩니다.

Undercurrents

언더커런트: 라이프사이클 전반을 떠받치는 원칙들

책에서 "undercurrent(저류)"라고 부르는 이 개념들은 특정 단계가 아니라 모든 단계에 걸쳐 적용되는 운영 원칙입니다. 파이프라인이 무너지거나 데이터가 신뢰를 잃는 대부분의 이유가 여기에 있습니다.

Security

보안과 접근 제어

더 많은 사람이 데이터를 쓸수록 더 강한 통제가 필요합니다. 역할 기반 접근 제어(RBAC), 컬럼 레벨 마스킹, 감사 로그가 핵심입니다. "개인정보가 들어 있는 컬럼은 마스킹 없이 분석가가 볼 수 없어야 한다"처럼 정책이 코드로 적용돼야 합니다.

Data Management

데이터 관리와 카탈로그

데이터의 뜻, 소유자, 최근 갱신 시점, 품질 점수를 문서화하고 검색할 수 있게 합니다. 데이터 카탈로그(Datahub, Alation)는 "이 테이블이 신뢰할 수 있는가? 누가 관리하는가?"를 빠르게 답해줍니다.

DataOps

데이터 운영 (DevOps의 데이터 버전)

파이프라인도 소프트웨어처럼 버전 관리, 테스트, 배포, 모니터링이 필요합니다. dbt 모델을 Git으로 관리하고, CI/CD 파이프라인에 품질 테스트를 포함시키고, 실패 시 알림을 보내는 것이 DataOps입니다.

Orchestration

오케스트레이션: 작업 순서 자동화

"먼저 A 테이블을 만들고, 성공하면 B를 계산하고, 실패하면 알림을 보낸다"처럼 의존 관계를 정의하고 스케줄을 관리합니다. Apache Airflow, Prefect, Dagster가 대표적입니다.

Architecture

아키텍처 선택

Lambda 아키텍처(배치+스트림 병렬), Kappa 아키텍처(스트림 단일화), Medallion 아키텍처(Bronze/Silver/Gold 레이어) 등 전체 설계 철학입니다. 성능, 비용, 운영 복잡도 트레이드오프가 결정됩니다.

Software Engineering

소프트웨어 엔지니어링 원칙

데이터 파이프라인 코드도 리뷰, 테스트, 문서화가 필요합니다. DRY(반복 금지), 모듈화, 명확한 인터페이스 설계는 데이터 코드에도 그대로 적용됩니다.

깊이 이해하기

Observability: 파이프라인의 건강 상태 모니터링

데이터 관측성(Data Observability)은 "데이터가 제대로 흐르고 있는가"를 실시간으로 파악하는 능력입니다. 소프트웨어의 APM(Application Performance Monitoring)과 유사합니다.

  • Freshness (신선도): 이 테이블은 마지막으로 언제 갱신됐는가? 6시간 전이라면 파이프라인이 실패했을 수 있습니다.
  • Volume (볼륨): 오늘 수집된 행 수가 평소와 비교해 급감하거나 급증했는가? 데이터 누락이나 중복의 신호입니다.
  • Distribution (분포): "status" 컬럼에 갑자기 새로운 값이 나타났는가? 소스 시스템이 바뀐 신호입니다.
  • Schema (스키마): 컬럼이 추가되거나 삭제됐는가? 하위 파이프라인이 깨질 수 있습니다.
  • Lineage (리니지): 이상이 발생했을 때 어떤 대시보드와 ML 모델이 영향을 받는가?
실제 사례: "어제 대시보드 숫자가 0이었는데 아무도 몰랐습니다"

파이프라인이 오전 4시에 실패했지만 알림이 없었고, 분석가가 오후 2시에 대시보드를 보고 처음 알아차렸습니다. 데이터 관측성 도구(Monte Carlo, Elementary 등)가 있었다면 볼륨이 0으로 떨어지는 즉시 Slack 알림이 갔을 것입니다.

Technology Stack

주요 기술 스택: 어떤 도구가 무슨 역할을 하나

도구를 외우는 것이 목표가 아닙니다. 각 도구가 라이프사이클의 어느 단계 문제를 해결하는지 이해하면, 새로운 도구를 만나도 빠르게 맥락을 잡을 수 있습니다.

라이프사이클 단계 대표 도구 무슨 문제를 해결하나

저장소 Snowflake, BigQuery, Databricks, Redshift 대규모 분석 쿼리를 빠르게 처리하는 클라우드 데이터 웨어하우스/레이크하우스입니다.

서빙 (BI) Looker, Tableau, Metabase, Superset 비코딩 사용자가 대시보드와 리포트를 만들고 볼 수 있게 합니다.

관측성 Monte Carlo, Elementary, Great Expectations 데이터 신선도, 볼륨, 분포 이상을 감지하고 알림을 보냅니다.

Infrastructure & Business Value

인프라 환경과 비즈니스 가치: 데이터 플랫폼은 어디서 돌아가고 왜 돈이 되는가

지금까지 배운 파이프라인은 결국 어딘가의 컴퓨터에서 실행됩니다. 그 "어딘가"가 자체 서버인지 클라우드인지에 따라 비용 구조와 운영 방식이 완전히 달라집니다. 그리고 이 모든 투자가 비즈니스에 어떻게 돌아오는지가 마지막 퍼즐입니다.

기초 개념

온프레미스 vs 클라우드: 서버를 사느냐, 빌리느냐

온프레미스(On-premise)는 회사가 직접 서버를 구매해 자체 전산실(데이터센터)에서 운영하는 방식이고, 클라우드는 AWS·GCP·Azure 같은 사업자의 컴퓨터를 필요한 만큼 빌려 쓰는 방식입니다.

비유: 자가 발전기 vs 전기 요금제

온프레미스는 집에 발전기를 설치하는 것입니다. 초기 비용이 크고 관리 인력이 필요하지만 통제권이 완전합니다. 클라우드는 전력회사에서 전기를 쓰는 만큼 사 오는 것입니다. 초기 비용이 거의 없고, 여름에 에어컨을 많이 틀면(트래픽 급증) 그만큼만 더 내면 됩니다.

비교 항목온프레미스클라우드
비용 구조CapEx(자본 지출) — 서버를 미리 구매OpEx(운영 지출) — 쓴 만큼 매달 지불
확장서버 주문 → 배송 → 설치 (수주~수개월)클릭 몇 번 또는 자동 (수분)
피크 대응최대 부하 기준으로 미리 구매 → 평소엔 놀림필요할 때만 늘리고 끝나면 줄임 (탄력성)
운영 인력하드웨어·네트워크·전원 관리 인력 필요물리 관리는 클라우드 사업자가 담당
적합한 경우규제로 데이터 외부 반출 불가, 예측 가능한 대규모 고정 부하스타트업~대기업 대부분의 데이터 워크로드
책의 관점

Fundamentals of Data Engineering은 현대 데이터 엔지니어링이 사실상 클라우드를 전제로 한다고 봅니다. 데이터 워크로드는 "새벽 2시 배치 때만 컴퓨팅이 몰리고 낮에는 한가한" 식으로 변동이 극심해서, 피크 기준으로 서버를 사두는 온프레미스 방식과 가장 안 맞는 워크로드이기 때문입니다.

핵심 개념

클라우드 서비스의 추상화 단계: IaaS → PaaS → SaaS → Serverless

클라우드라고 다 같은 클라우드가 아닙니다. "어디까지 직접 관리하고 어디부터 맡기느냐"의 스펙트럼이 있습니다.

IaaS

가상 서버만 빌리기 (EC2, GCE)

빈 컴퓨터를 빌려서 OS 설치부터 직접 합니다. 자유도가 가장 높지만 패치, 보안, 확장을 모두 직접 관리해야 합니다. "전셋집을 빌려 인테리어부터 직접"에 가깝습니다.

PaaS

플랫폼까지 제공 (RDS, BigQuery)

DB나 실행 환경이 설치·관리된 상태로 제공됩니다. 백업, 패치, 복제를 사업자가 처리합니다. 데이터 팀은 테이블과 쿼리에만 집중합니다.

SaaS

완성된 소프트웨어 구독 (Fivetran, Looker)

설치 자체가 없습니다. 브라우저로 로그인해서 바로 씁니다. 수집 도구, BI 도구 대부분이 이 형태입니다.

Serverless

서버 개념 자체가 사라짐 (Lambda, BigQuery 쿼리 과금)

코드나 쿼리가 실행되는 순간에만 자원이 할당되고 그 시간만큼만 과금됩니다. "서버가 없는" 게 아니라 "서버를 의식할 필요가 없는" 것입니다. 하루 한 번 5분 도는 파이프라인을 위해 24시간 서버를 켜둘 필요가 없어집니다.

⚠️ 책이 강조하는 선택 기준: 미분화된 중노동을 피하라

책은 "undifferentiated heavy lifting(차별화되지 않는 중노동)"이라는 표현을 씁니다. 서버 패치, Kafka 클러스터 직접 운영 같은 일은 아무리 잘해도 회사의 경쟁력이 되지 않습니다. 경쟁력과 무관한 레이어는 관리형 서비스에 맡기고, 팀의 시간은 비즈니스에 직결되는 데이터 모델과 품질에 쓰라는 것이 일관된 조언입니다.

핵심 개념

컴퓨트와 스토리지의 분리: 현대 데이터 플랫폼의 결정적 구조

전통적 DB는 저장(디스크)과 계산(CPU)이 한 몸이라 저장 용량을 늘리려면 계산 능력도 같이 사야 했습니다. Snowflake, BigQuery 같은 현대 플랫폼은 둘을 분리했습니다. 데이터는 저렴한 오브젝트 스토리지에 한 벌만 두고, 계산 자원은 필요할 때 필요한 만큼 띄웁니다.

비유: 도서관과 열람실

책(데이터)은 서고에 한 벌만 보관하고, 열람실(컴퓨트)은 이용자가 많을 때 여러 개 열고 새벽엔 닫습니다. 재무팀 열람실과 분석팀 열람실을 따로 열면 서로 자리 경쟁(쿼리 간섭)도 없습니다.

  • 독립 확장: 데이터가 10배 늘어도 쿼리가 적으면 컴퓨트 비용은 그대로입니다. 반대도 마찬가지입니다.
  • 워크로드 격리: 월말 정산 배치, 임원 대시보드, 분석가의 무거운 실험 쿼리를 서로 다른 컴퓨트에서 돌려 간섭을 없앱니다.
  • 유휴 비용 제로: 쿼리가 없으면 컴퓨트를 꺼서 스토리지 비용(매우 저렴)만 냅니다.
전체 그림

기존 서비스와 데이터 플랫폼은 어떻게 공존하는가

데이터 플랫폼은 기존 서비스를 대체하는 것이 아니라 그 옆에 나란히 서서, 서비스가 만들어내는 데이터를 받아 가공하고 다시 서비스와 사람에게 돌려주는 순환 구조를 만듭니다.

기존 서비스앱 서버 + 운영 DB가 평소처럼 주문·결제 처리
수집 계층CDC·이벤트 스트림이 운영에 부담 없이 데이터 복제
데이터 플랫폼클라우드 웨어하우스에서 저장·변환·품질 검사
소비 계층대시보드, ML 모델, Reverse ETL
다시 서비스로추천·이탈 점수가 앱과 CRM에 반영 → 새 데이터 생성
실제 예시: 음식 배달 서비스의 하루

① 낮 동안 주문 서비스는 운영 DB에 주문을 기록합니다(데이터 플랫폼은 전혀 개입하지 않음). ② CDC가 주문 변경분을 실시간으로 웨어하우스에 복제합니다. ③ 새벽 배치가 "지역별 수요 예측" 모델용 데이터를 만들고, 모델이 다음 날 라이더 배치 추천을 계산합니다. ④ 그 결과가 운영 시스템으로 역전송되어 아침부터 배차에 반영됩니다. 운영 서비스와 데이터 플랫폼은 서로 다른 인프라에서 돌지만, 데이터로 연결된 하나의 순환 고리입니다.

⚠️ 분리의 핵심 이유: 장애 격리

분석 쿼리가 아무리 폭주해도 고객의 주문·결제는 영향받지 않아야 합니다. 운영 환경과 분석 환경을 물리적으로 분리하는 것은 성능 문제이기 이전에 "분석 때문에 서비스가 죽는 일"을 원천 차단하는 안전 설계입니다.

비즈니스 가치

데이터 성숙도 3단계: 데이터가 돈이 되기까지

책은 조직이 데이터로 가치를 만드는 과정을 3단계 성숙도 모델로 설명합니다. 지금 우리 조직이 어느 단계인지 알면, 무엇에 먼저 투자해야 하는지가 보입니다.

1단계
Starting

데이터로 시작하기 — 일단 모으고 신뢰를 만든다

데이터가 흩어져 있고 리포트는 수작업입니다. 이 단계의 목표는 화려한 AI가 아니라 "핵심 데이터를 한곳에 모으고, 기본 리포트가 맞는 숫자를 내게 하는 것"입니다. 책은 이 단계에서 ML부터 시작하는 것을 흔한 실패 패턴으로 꼽습니다 — 신뢰할 수 있는 데이터 기반 없이 모델을 만들면 모델도 신뢰할 수 없기 때문입니다.

2단계
Scaling

데이터로 확장하기 — 사람이 아니라 시스템이 일하게 한다

파이프라인이 자동화되고, 품질 테스트와 모니터링이 갖춰지고, 여러 팀이 셀프서비스로 데이터를 씁니다. 데이터 팀이 "리포트 뽑아주는 팀"에서 "데이터 제품을 운영하는 팀"으로 바뀌는 단계입니다.

3단계
Leading

데이터로 앞서가기 — 데이터가 제품과 수익의 일부가 된다

ML이 핵심 업무 흐름에 들어가고(추천, 사기 탐지, 수요 예측), 데이터 자체가 제품이 되기도 합니다. 이 단계는 1·2단계의 탄탄한 기반 위에서만 가능합니다.

비즈니스 가치

그래서 비즈니스에 정확히 어떻게 도움이 되는가

데이터 플랫폼 투자의 수익은 크게 네 갈래로 돌아옵니다. 각각 실제 업무 장면과 연결해 보면 추상적인 "데이터 기반 의사결정"이 구체화됩니다.

가치 유형전 (플랫폼 없음)후 (플랫폼 있음)
의사결정 속도 "지난주 가입 추이 좀 뽑아주세요" → 데이터 팀 티켓 → 3일 대기 현업이 대시보드에서 즉시 확인. 회의가 "숫자 검증"이 아니라 "결정"으로 시작
운영 비용 절감 매월 재무 마감 때 여러 시스템 엑셀을 수작업 대조 (며칠 소요) 정합성이 검증된 마트에서 자동 집계. 사람은 예외 케이스만 확인
매출 기여 이탈 고객을 떠난 뒤에 알게 됨 이탈 위험 점수가 CRM에 매일 동기화 → CS가 선제 대응 → 리텐션 상승
리스크 감소 "고객 개인정보를 누가 봤는지" 감사 요청에 답할 수 없음 접근 제어와 감사 로그가 플랫폼에 내장 → 규제 대응 비용 감소
책의 핵심 관점: 데이터 엔지니어링은 비용이 아니라 지렛대

잘 만든 파이프라인 하나가 분석가 열 명의 반복 작업을 없애고, 잘 정의된 마트 하나가 회사 전체의 "숫자 논쟁"을 끝냅니다. 데이터 엔지니어링의 가치는 그 위에서 일하는 모든 사람의 생산성에 곱해지는(multiply) 방식으로 실현됩니다. 반대로 기반이 부실하면 그 위의 모든 분석과 ML이 모래성이 됩니다.

⚠️ 단, 클라우드 비용은 저절로 줄지 않는다 (FinOps)

"쓴 만큼 낸다"는 "아낀 만큼 덜 낸다"이기도 하지만 "실수한 만큼 더 낸다"이기도 합니다. 조건 없는 SELECT *로 전체 테이블을 스캔하는 쿼리, 끄는 것을 잊은 컴퓨트, 무한히 쌓이는 미사용 데이터가 청구서를 키웁니다. 그래서 비용을 팀별·워크로드별로 추적하고 최적화하는 FinOps가 현대 데이터 조직의 필수 역량이 되었습니다.
